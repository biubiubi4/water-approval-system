# 会话工作总结与合规升级计划

本文档总结截至 2026-07-13 本轮会话围绕“取水申请材料合规性检测系统”完成的主要工作，并规划下一阶段将 `_check_compliance()` 从简单兜底规则升级为“合规维度清单 + RAG 条款依据 + Qwen 复核”的实施路径。

## 一、本轮会话完成内容

### 1. 审核调用链路与边界梳理

当前系统调用边界已统一为：

```text
Vue 前端 -> Java Spring Boot 后端 -> Python AI 服务
```

- 前端只通过 `/api/**` 访问 Java 后端。
- Java 后端负责申请保存、附件保存、审核触发和审核结果持久化。
- Python AI 服务负责附件解析、规则审查、知识库检索、本地合规兜底和 Qwen 按需复核。

### 2. 规则前置与审核加速

已新增并完善 `python-ai-service/app/review_rules.py`，将确定性规则前置，用于快速处理无需大模型判断的问题。

当前规则能力包括：

- 必填字段检查：申请人或申请单位、证件号或统一社会信用代码、项目名称、取水用途、取水地点。
- 必备附件检查：取水许可申请书、身份证或营业执照。
- 标准申请书结构识别：支持识别“申请人基本情况”“项目基本情况”“运行期年取水量”等标准申请书字段。
- 附件类型一致性检查：文件名疑似身份证、营业执照、申请书时，结合解析文本判断是否明显不匹配。
- 有效期检查：识别附件文本中的证件或资质过期信息。
- 表单与附件一致性检查：申请人名称、证件号是否能在附件文本中找到。

已去除“水资源论证相关材料”的必备要求，不再因为缺少该材料产生缺材料提醒或导致材料完整性异常。

### 3. 统一完整性规则来源

已将材料完整性结果统一为同一套规则来源：

```text
evaluate_application_rules()
-> build_completeness_from_rules()
-> details.fast_rules / details.completeness
```

关键改动：

- `details.fast_rules` 和 `details.completeness` 均来自解析后的同一份 `full_rules`。
- `tools.check_completeness()` 改为包装 `evaluate_application_rules()`，不再单独维护一套缺字段、缺材料逻辑。
- `agent.py` 主审核流程不再调用 `check_completeness` 二次计算完整性。
- 审核 trace 中新增规则版本：`2026-07-13-standard-application-form-v3`。

这样可以避免“审核结论通过，但材料完整性仍显示不完整”的前后矛盾。

### 4. 附件解析缓存

已新增 `python-ai-service/app/document_cache.py`，通过文件 SHA256 缓存附件解析结果。

实现效果：

- 审核结果中返回 `details.document_cache`，展示缓存启用、命中片段数、未命中片段数和文件维度统计。
- 文档解析缓存已迁移至 `E:\\water-approval-runtime\\document_cache/`，不再位于项目工作区，因此不会被 Git 跟踪。

## 二、当前审核逻辑摘要

当前审核流程为：

```text
申请数据 + 附件路径
-> 附件解析或读取解析缓存
-> evaluate_application_rules 规则审查
-> build_completeness_from_rules 生成材料完整性
-> 知识库检索
-> 按 review_mode 判断是否调用 Qwen
-> 合并本地合规结果或 Qwen 复核结果
-> 返回统一审核结果
```

其中：

- `BLOCKER` 表示硬性问题，会直接驳回。
- `WARNING` 表示需要复核，不应单独导致审核失败。
- `details.fast_rules` 是规则命中详情。
- `details.completeness` 是从同一份规则结果生成的材料完整性视图。
- `details.ai_review_trace.rule_version` 用于确认规则版本。

## 三、当前合规性判断的不足

当前 `_check_compliance()` 仍然较简单，只适合作为兜底逻辑。它主要检查：

- 是否缺少 `project_name`、`applicant_name`、`applicant_id`。
- `water_use` 中是否包含“饮用水”等粗粒度风险词。
- 知识库检索命中内容中是否包含“禁止”。

不足之处：

- 没有形成稳定的合规维度清单。
- 没有把标准申请书中的取水量、水源类型、计量方式、退水方式、期限等字段结构化。
- 没有将 RAG 命中的法规条款绑定到具体合规维度。
- Qwen 复核结果缺少固定 JSON 输出约束，难以稳定展示和追踪。
- 合规结论的解释性还不够强，不利于页面展示和答辩说明。

## 四、下一阶段目标

下一阶段目标是将 `_check_compliance()` 升级为：

```text
合规维度清单 + RAG 条款依据 + Qwen 复核
```

期望效果：

- 本地规则先给出结构化合规检查项。
- RAG 检索结果能绑定到具体检查维度。
- Qwen 只处理复杂语义判断和法规适用判断。
- 返回结果可直接用于前端展示“检查项、结论、依据、建议”。

## 五、实施计划

### 阶段 1：定义合规维度清单

新增或扩展合规规则结构，建议文件：

```text
python-ai-service/app/compliance_rules.py
```

建议维度：

| 维度 | 检查内容 | 结果等级 |
| --- | --- | --- |
| applicant_identity | 申请主体、证件号、营业执照信息是否一致 | BLOCKER / WARNING / PASS |
| application_form | 标准申请书关键栏目是否完整 | BLOCKER / WARNING / PASS |
| water_source | 水源类型、取水地点、取水口位置是否完整合理 | WARNING / PASS |
| water_quantity | 年取水量、施工期取水量是否填写且格式合理 | WARNING / PASS |
| water_use | 取水用途是否明确，是否存在限制性用途 | WARNING / PASS |
| metering | 计量方式和数据传输方式是否填写 | WARNING / PASS |
| drainage | 年退水量、退水方式、排放去向是否填写合理 | WARNING / PASS |
| commitment | 承诺文本、承诺人签章信息是否存在 | WARNING / PASS |
| legal_restriction | 是否命中禁止、限制、不予批准等法规条款 | BLOCKER / WARNING / PASS |

输出结构建议：

```json
{
  "pass": true,
  "status": "PASS",
  "dimensions": [
    {
      "code": "water_source",
      "name": "水源与取水地点",
      "status": "PASS",
      "findings": [],
      "evidence": [],
      "suggestions": []
    }
  ],
  "violations": [],
  "warnings": []
}
```

### 阶段 2：抽取标准申请书字段

在附件解析后，对标准申请书文本做轻量结构化抽取。

建议新增：

```text
python-ai-service/app/application_form_parser.py
```

抽取字段包括：

- 申请人或申请单位
- 统一社会信用代码或身份证号码
- 法定代表人
- 项目名称
- 项目性质
- 项目概况
- 运行期年取水量
- 水源类型
- 取水地点
- 取水口位置
- 年取水量
- 取水工程类型
- 申请事由
- 申请取水起始时间
- 期限
- 取水用途
- 计量方式
- 年退水量
- 退水方式和排放去向
- 承诺内容

抽取结果放入：

```json

  "details": {
    "application_form_fields": {}
  }
}
```

### 阶段 3：升级本地合规判断

将 `_check_compliance()` 拆分为维度化检查：

```text
_check_compliance()
-> evaluate_compliance_dimensions()
-> merge_compliance_findings()
```

本地规则负责确定性判断，例如：

- 明确缺少取水地点，给 `WARNING`。
- 明确选择“其他”但没有具体说明，给 `BLOCKER` 或 `WARNING`。
- 明确缺少计量方式，给 `WARNING`。
- 年退水量为空但不是无退水场景，给 `WARNING`。
- 知识库命中禁止性条款，给 `WARNING` 并交由 Qwen 复核。

### 阶段 4：RAG 按维度检索条款

当前知识库检索是单一 query，后续应按合规维度构造 query。

示例：

```text
取水地点 + 保护区 + 取水许可 + 禁止
取水用途 + 工业用水 + 取水许可 + 限制
取水量 + 水资源论证 + 管控指标
退水排放去向 + 水功能区 + 取水许可
计量设施 + 取水计量 + 管理要求
```

每个维度保留命中的法规依据：

```json
{
  "dimension": "legal_restriction",
  "hits": [
    {
      "source": "法规文件名",
      "content": "条款摘要",
      "score": 0.82
    }
  ]
}
```

### 阶段 5：Qwen 结构化复核

当满足以下条件时调用 Qwen：

- 命中限制性法规条款。
- 本地规则给出多个 WARNING。
- 附件文本较长或字段冲突。
- `review_mode = strict`。

要求 Qwen 返回固定 JSON：

```json
{
  "status": "APPROVED",
  "message": "合规性复核通过",
  "compliance": {
    "pass": true,
    "dimensions": [
      {
        "code": "water_use",
        "status": "PASS",
        "reason": "取水用途为工业用水，未命中禁止性条款。",
        "legal_basis": []
      }
    ],
    "violations": [],
    "suggestions": []
  }
}
```

注意事项：

- Qwen 只能补充语义判断，不应覆盖本地 `BLOCKER`。
- Qwen 输出必须做 JSON 解析和字段校验。
- Qwen 不可用时，应回退本地维度化合规结果。

### 阶段 6：统一结果结构与前端展示

最终返回结构建议：

```json
{
  "status": "APPROVED",
  "message": "申请审核通过",
  "suggestions": [],
  "knowledge_hits": [],
  "details": {
    "fast_rules": {},
    "completeness": {},
    "compliance": {
      "pass": true,
      "status": "PASS",
      "dimensions": [],
      "violations": [],
      "warnings": []
    },
    "rag_evidence": {},
    "external_ai": {},
    "ai_review_trace": {}
  }
}
```

前端建议展示：

- 合规总览：通过 / 有风险 / 不通过。
- 维度清单：每个维度显示状态、问题、建议。
- 法规依据：展示命中的法规来源和摘要。
- AI 复核：展示是否调用 Qwen、调用原因、模型结论。

## 六、建议优先级

建议按以下顺序推进：

1. 新增 `compliance_rules.py`，定义合规维度和本地维度化检查。
2. 改造 `_check_compliance()`，返回 `dimensions` 结构。
3. 在 `agent.py` 中保留现有审核流程，只替换 compliance 生成逻辑。
4. 按维度增强 RAG query，保存 `details.rag_evidence`。
5. 约束 Qwen 输出 JSON，并将结果写入 `details.external_ai.compliance`。
6. 前端新增合规维度展示组件。

## 七、验收标准

### 后端验收

- 材料完整性仍只由 `fast_rules` 生成。
- 合规性结果包含 `dimensions`。
- 每个维度有 `code`、`name`、`status`、`findings`、`suggestions`。
- 命中法规时能在 `rag_evidence` 中看到来源。
- Qwen 不可用时仍能返回本地合规结果。
- 本地 `BLOCKER` 不会被 Qwen 覆盖为通过。

### 前端验收

- 审核详情能看到合规维度列表。
- 能区分“材料不完整”和“合规性风险”。
- 能看到法规依据和 AI 复核状态。
- 旧审核结果没有新字段时，页面仍能兼容展示。

### 可解释性验收

- 每个不通过或警告项都能说明原因。
- 每个法规风险都能追溯到知识库命中条款。
- 每次审核都能通过 `rule_version` 和 `qwen_decision_reason` 追踪路径。

## 八、注意事项

- 不要把材料完整性和合规性混成一个结果。
- `WARNING` 只表示需要复核，不应直接导致驳回。
- `BLOCKER` 是硬性问题，可以直接驳回。
- 附件解析缓存只缓存文本，不缓存规则判断。
- 规则版本变化后，已有旧审核结果需要重新审核才会更新。
