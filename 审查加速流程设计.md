# 审查加速流程设计

本文档用于固化取水申请材料智能审查的后续开发方案。目标是在保证审查质量的前提下，减少无效大模型调用、缩短审核等待时间，并让审查结论更稳定、可解释。

## 一、设计目标

当前系统已经具备以下能力：

- 前端提交申请与附件。
- Java 后端保存申请、附件和审核结果。
- Python AI 服务执行附件解析、完整性检查、知识库检索、Qwen 文本审查和本地规则兜底。

后续优化目标：

- 前置确定性规则，快速发现明显不合规问题。
- 降低 Qwen 文本审查和 Qwen-VL PDF 识别的调用次数。
- 保留 RAG 与大模型对复杂材料理解的优势。
- 输出可追踪的审查过程，方便页面展示、日志排查和答辩说明。

推荐采用：

```text
申请数据与附件 -> 快速规则审查 -> 附件解析缓存 -> 知识库检索 -> Qwen 按需复核 -> 统一审核结果
```

## 二、总体调用边界

系统调用边界保持为：

```text
Vue 前端 -> Java 主后端 -> Python AI 服务
```

- 前端只调用 Java 后端 `/api/**`。
- Java 后端负责编排业务流程、文件保存、结果持久化。
- Python AI 服务负责智能审查能力，不直接暴露给前端。

不要在前端开发代理或页面代码中直接访问 Python 服务端口。

## 三、推荐审查流程

### 1. 创建申请

触发时机：用户在前端提交申请表单。

流程：

```text
Vue 表单
-> Java /api/applications
-> 保存申请字段
-> 保存上传附件
-> 返回申请记录
```

这一阶段不调用 Qwen。它只负责收集材料和保存业务数据。

### 2. 触发审核

触发时机：用户点击“开始审核”或“重新审核”。

流程：

```text
Vue
-> Java /api/applications/{id}/review
-> Java 组装申请数据和附件路径
-> Python /api/review
```

Python 审核流程建议拆成以下步骤：

```text
1. 快速规则审查
2. 附件解析与缓存读取
3. 材料完整性检查
4. 知识库语义检索
5. 判断是否需要调用 Qwen
6. 合并规则、检索、大模型结果
7. 返回统一审核结果
```

## 四、快速规则审查

快速规则审查用于处理明确、稳定、无需大模型理解的问题。

建议新增文件：

```text
python-ai-service/app/review_rules.py
```

建议包含以下规则类型：

- 必填字段规则。
- 必备附件规则。
- 申请书填写规则。
- 附件类型匹配规则。
- 证件有效期规则。
- 申请数据与附件文本一致性规则。
- 高风险场景规则。

### 4.1 必填字段规则

示例字段：

- `applicant_name`
- `applicant_id`
- `project_name`
- `water_use`
- `location`

如果缺少关键字段，可直接输出问题项。

### 4.2 必备附件规则

基础必备材料：

- 取水许可申请书。
- 身份证或营业执照。
- 水资源论证相关材料。

可根据申请主体类型区分：

- 个人：身份证明。
- 企业：营业执照或统一社会信用代码证。

### 4.3 申请书填写规则

推荐优先固化的申请书规则：

- 项目名称不能为空。
- 申请人或申请单位不能为空。
- 取水用途不能为空。
- 取水地点不能为空。
- 联系方式不能为空。
- 如果取水用途选择“其他”，必须填写具体用途说明。
- 申请表中关键字段不能出现空白、未填、无、待补充等占位文本。

### 4.4 附件类型匹配规则

文件名和解析文本需要相互印证。

示例：

- 文件名包含“身份证”，但解析文本中出现“驾驶证”，应判定材料类型不匹配。
- 文件名包含“营业执照”，但解析文本缺少“统一社会信用代码”“营业期限”等关键字，应标记为疑似不匹配。
- 文件名包含“申请书”或“申请表”，但解析文本缺少“取水许可申请”相关表头，应标记为疑似不匹配。

### 4.5 有效期规则

需要从附件解析文本中识别：

- 身份证有效期。
- 营业执照营业期限。
- 其他资质文件有效期。

如果有效期早于当前日期，直接判定不合格。

### 4.6 一致性规则

申请表单数据应与附件解析文本一致。

重点校验：

- 申请人姓名。
- 身份证号或统一社会信用代码。
- 项目名称。
- 申请单位名称。

如果字段不一致，输出具体冲突项。

## 五、硬性驳回与 Qwen 按需调用

建议将规则结果分为三类：

```text
BLOCKER  硬性驳回，不需要调用 Qwen
WARNING  存疑问题，可调用 Qwen 复核
PASS     规则通过
```

### 5.1 直接驳回场景

以下情况建议不调用 Qwen，直接返回 `REJECTED`：

- 未上传任何附件。
- 缺少取水许可申请书。
- 缺少身份证明或营业执照。
- 必填字段缺失。
- 证件明确过期。
- 附件类型明显错误。

### 5.2 需要 Qwen 复核场景

以下情况建议调用 Qwen：

- 附件文本较长，需要综合理解。
- 附件内容与申请数据可能不一致，但规则无法完全确认。
- 知识库检索命中限制性法规。
- 用户上传了复杂 PDF 表格、扫描件或多页材料。
- 审查模式设置为严格模式。

## 六、审查模式

建议在 Python 配置中加入审查模式：

```text
review_mode = smart
```

可选值：

```text
FAST    只执行规则审查、附件解析和知识库检索，不调用 Qwen 文本审查
SMART   默认模式，硬性问题直接驳回，复杂问题调用 Qwen
STRICT  始终调用 Qwen 复核，适合演示或高风险材料
```

建议默认使用 `SMART`。

## 七、附件解析缓存

附件解析是耗时重点，尤其是扫描件 PDF 触发 Qwen-VL 时。

建议新增缓存目录：

```text
python-ai-service/data/document_cache/
```

缓存 key 建议使用：

```text
文件绝对路径 + 文件大小 + 文件修改时间
```

或使用更稳的：

```text
文件 SHA256
```

缓存内容建议包含：

- `file_path`
- `file_hash`
- `reader`
- `pages`
- `chunks`
- `created_at`
- `updated_at`

命中缓存时，直接读取解析文本，不重复调用 PDF 解析或 Qwen-VL。

## 八、统一审核结果结构

建议 Python `/api/review` 返回结构保持统一：

```json
{
  "status": "APPROVED",
  "message": "申请审核通过",
  "suggestions": [],
  "knowledge_hits": [],
  "details": {
    "ai_review_trace": {},
    "fast_rules": {},
    "completeness": {},
    "compliance": {},
    "external_ai": {},
    "attachment_documents": []
  }
}
```

字段说明：

- `status`：`APPROVED`、`REJECTED`、`PENDING`、`ERROR`。
- `message`：审核结论摘要。
- `suggestions`：面向用户的修改建议。
- `knowledge_hits`：知识库检索结果。
- `details.ai_review_trace`：是否调用 Qwen、使用模型、是否回退本地规则。
- `details.fast_rules`：快速规则审查结果。
- `details.completeness`：材料完整性检查结果。
- `details.compliance`：本地合规规则结果。
- `details.external_ai`：Qwen 返回的补充推理。
- `details.attachment_documents`：附件解析片段。

## 九、建议代码落点

### 9.1 Python AI 服务

建议新增：

```text
python-ai-service/app/review_rules.py
python-ai-service/app/document_cache.py
```

建议改造：

```text
python-ai-service/app/agent.py
python-ai-service/app/tools.py
python-ai-service/app/documents.py
python-ai-service/app/config.py
```

职责建议：

- `review_rules.py`：集中维护申请书规则和硬性驳回规则。
- `document_cache.py`：维护附件解析缓存。
- `agent.py`：编排规则、解析、检索、Qwen、结果合并。
- `tools.py`：复用规则配置进行完整性检查。
- `documents.py`：读取缓存或写入缓存。
- `config.py`：增加审查模式、缓存开关等配置。

### 9.2 Java 后端

建议保持现有职责：

- 保存申请。
- 保存附件。
- 触发审核。
- 保存最新审核结果与历史审核结果。
- 不在 Java 中直接写复杂 AI 规则。

可后续增加：

```text
reviewMode
```

用于让前端选择 `FAST`、`SMART`、`STRICT`。

### 9.3 前端

建议后续增加：

- 审查模式选择。
- 审查耗时展示。
- 是否调用 Qwen 的标识展示。
- 规则命中问题列表。
- 附件解析来源展示，例如普通 PDF 解析或 Qwen-VL 识别。

## 十、推荐实施顺序

### 第一阶段：规则前置

目标：快速减少无效 Qwen 调用。

任务：

1. 新增 `review_rules.py`。
2. 将必填字段、必备材料、申请书填写规则移入规则模块。
3. 在 `agent.py` 中先执行快速规则审查。
4. 遇到 `BLOCKER` 直接返回 `REJECTED`。

验收：

- 缺少申请书时，不调用 Qwen。
- 缺少身份证或营业执照时，不调用 Qwen。
- 必填字段为空时，不调用 Qwen。
- 返回结果中包含 `details.fast_rules`。

### 第二阶段：Qwen 按需调用

目标：让 Qwen 只处理复杂判断。

任务：

1. 在 `config.py` 中增加 `review_mode`。
2. 实现 `FAST`、`SMART`、`STRICT` 三种模式。
3. 在 `details.ai_review_trace` 中记录模式和调用结果。

验收：

- `FAST` 模式不调用 Qwen 文本审查。
- `SMART` 模式在硬性问题时不调用 Qwen。
- `STRICT` 模式始终调用 Qwen 文本审查。

### 第三阶段：附件解析缓存

目标：减少重复 PDF 解析和 Qwen-VL OCR。

任务：

1. 新增 `document_cache.py`。
2. 在 `documents.py` 中接入缓存读取和写入。
3. 在日志和返回结果中标明是否命中缓存。

验收：

- 同一附件第二次审核时不重复调用 Qwen-VL。
- 返回结果中可以看到附件解析来源和缓存命中状态。

### 第四阶段：前端可视化

目标：让用户理解审核过程。

任务：

1. 在审核结果页展示规则命中项。
2. 展示是否调用 Qwen。
3. 展示审查模式。
4. 展示附件解析方式。

验收：

- 页面可以看到“规则审查”“知识库检索”“AI 复核”三个阶段结果。

## 十一、日志规范

建议统一使用以下日志前缀：

```text
[规则审查]
[附件解析]
[PDF解析]
[知识检索]
[AI审核]
[审核结果]
```

禁止打印：

- API Key。
- 用户敏感证件完整号码。
- 大段附件全文。

允许打印：

- 是否启用外部 AI。
- 模型名称。
- 是否调用成功。
- 是否回退本地规则。
- 命中的规则编号。
- 附件解析方式。
